GUERRES ENTRE LES NAVIGATEURS

L'une des raisons pour lesquelles les concepteurs Web n'utilisaient pas CSS était le manque de prise en charge du navigateur. À l'époque, il y avait deux principaux navigateurs en compétition pour l'âme du Web : Microsoft Internet Explorer et Netscape Navigator. Ils étaient incompatibles par conception. Un navigateur inventerait un nouvel élément ou attribut HTML. L'autre navigateur inventerait son propre élément ou attribut différent mais pour faire exactement la même chose. Peut-être que l'idée derrière cette stratégie était que les concepteurs de sites Web auraient à choisir les fonctionnalités propriétaires qu'ils allaient adopter, de la même manière qu’un enfant étant forcés de choisir entre ses parents. En réalité, les concepteurs Web n'avaient guère d'autre choix que d'écrire pour les deux navigateurs, ce qui signifiait faire deux fois plus de travail. Un groupe de concepteurs Web a décidé que cela suffisait. Ils se sont réunis sous la bannière du Web Standards Project et ont commencé à faire pression sur Microsoft et Netscape pour qu'ils abandonnent leurs méthodes propriétaires et adoptent des normes telles que CSS. Le vent a commencé à tourner avec le lancement d'Internet Explorer 5 pour Mac, un navigateur livré avec un support CSS impressionnant. Si tel était l'avenir de la conception Web, la vie était sur le point de devenir beaucoup plus productive et créative. Certains concepteurs Web avant-gardistes ont détecté le bug CSS tôt. Ils ont repensé leurs sites Web en utilisant CSS pour la mise en page au lieu d'utiliser des TABLEAUX et des GIF d'espacement. Fidèles à l'esprit fondateur du Web, ils ont partagé ce qu'ils apprenaient et ont encouragé les autres à passer au CSS. La meilleure démonstration de la puissance du CSS était peut-être un site Web appelé CSS Zen Garden, créé par Dave Shea. C'était une vitrine de designs magnifiques et variés, tous réalisés avec CSS. Fondamentalement, le HTML est resté le même. Voir le même document HTML stylisé d'une multitude de manières différentes a conduit à l'un des effets bénéfiques du CSS: separation of concerns.

COUPLAGE

Dans n'importe quel système, de l'infrastructure urbaine à un programme informatique, les concepteurs de ce système peuvent choisir dans quelle mesure les éléments du système dépendent les uns des autres. Dans un système très couplé, chaque pièce dépend de chaque pièce. Dans un système faiblement couplé, toutes les pièces sont indépendantes, avec peu ou pas de connaissance des autres pièces. Dans un système très couplé, chaque partie du système peut faire des hypothèses sur les autres parties. Ces systèmes peuvent être conçus assez rapidement, mais à un certain prix. Il manque de résilience. Si une pièce échoue, elle pourrait faire échouer tout le système avec elle. La conception d'un système faiblement couplé peut demander plus de travail. L’avantage est que le résultat global est plus résistant à l'échec. Les différentes parties du système peuvent être remplacées avec un minimum d'effets d'entraînement. Les hacks mis au point par David Siegel ont faiblement couplé la structure et la présentation dans un seul fichier HTML monolithique. L'adoption de CSS a facilité cette dépendance, rapprochant le Web de l'approche modulaire de la philosophie UNIX. Les informations de présentation pourraient être déplacées dans un fichier séparé: une feuille de style. C’est ainsi qu’un seul document HTML du CSS Zen Garden pourrait avoir autant de conceptions différentes. La feuille de style doit encore avoir une certaine connaissance de la structure du document HTML. Assez souvent, des «hooks» sont ajoutés dans le balisage pour faciliter le style: des valeurs spécifiques d'attributs CLASS ou ID, par exemple. Ainsi, HTML et CSS ne sont pas complètement découplés. Ils forment un partenariat mais ils ont aussi un arrangement. Le document de balisage peut décider qu'il souhaite parfois voir d'autres feuilles de style. Pendant ce temps, la feuille de style pourrait éventuellement être appliquée à d'autres documents. Ils sont faiblement couplés.

DANSER SUR L'ARCHITECTURE

Il faut du temps à une discipline pour développer ses propres valeurs de conception. La conception Web est en effet une jeune discipline. Alors que nous commençons lentement à former notre propre ensemble de principes directeurs, nous pouvons nous tourner vers d'autres disciplines pour nous inspirer. Le monde de l'architecture a accumulé ses propres valeurs de conception au fil des ans. L'une de ces valeurs est le principe de l'honnêteté matérielle. Un matériau ne doit pas être utilisé comme un substitut à un autre. Sinon, le résultat final est trompeur. L'utilisation de TABLE pour la mise en page est matériellement malhonnête. L'élément TABLE est destiné à baliser la structure des données tabulaires. Le résultat final de l'utilisation des TABLEAUX, des éléments FONT et des GIF d'espacement est une façade. À première vue, tout semble bien, mais il ne résiste pas à un examen minutieux. Dès qu'un tel site Web est testé en fonction de son utilisation réelle sur une gamme de navigateurs, la façade s'effrite. L’utilisation de CSS pour la présentation est matériellement honnête - c’est l’utilisation prévue de CSS. Cela permet également au HTML d'être matériellement honnête. Au lieu d'essayer de remplir deux rôles - la structure et la présentation - le HTML peut revenir à remplir son véritable objectif, en balisant la signification du contenu. Il est toujours possible d'utiliser (ou d'abuser) du CSS pour être matériellement malhonnête. Pendant très longtemps, il n'y avait pas de moyen facile d'ajouter des coins arrondis à un élément en utilisant CSS. Au lieu de cela, les concepteurs Web ont trouvé des moyens de contourner le problème, en plaçant des images d'arrière-plan sur l'élément pour simuler le même effet final. Cela a fonctionné jusqu'à un certain point, mais tout comme le hack GIF spacer, c'était une façade. Puis la propriété border-radius est arrivée. Désormais, les concepteurs peuvent avoir leurs coins arrondis d'une manière matériellement honnête. De manière cruciale, les concepteurs ont pu utiliser de nouvelles propriétés telles que le rayon de la bordure bien avant que chaque navigateur Web ne les prenne en charge. Tout cela grâce au modèle libéral de gestion des erreurs de CSS. Les navigateurs plus récents afficheraient les coins arrondis. Les navigateurs plus anciens ne généreraient pas d'erreur. Les navigateurs plus anciens n'arrêteraient pas d'analyser le CSS et refusaient d'analyser davantage. Ils ignoreraient simplement les instructions qu'ils ne comprenaient pas et passeraient à autre chose. Pas de mal, pas de faute. Bien sûr, cela signifie que le site Web résultant sera différent dans différents navigateurs. Certaines personnes verront des coins arrondis. D'autres non. Et ça va.
